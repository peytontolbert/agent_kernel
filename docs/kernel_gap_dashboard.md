## Current Read

The kernel already has the pieces of a serious coding-ASI runtime:

- bounded task execution
- verifier-driven decision closeout
- memory, world-model, and governance paths in the live runtime
- candidate-vs-baseline self-improvement closeout
- unattended campaign orchestration

The main blocker is no longer "missing basic machinery."

The blocker for the stronger repeated/integrated claim is weak conversion from
those existing capabilities into:

- repeatable retained gains
- counted cross-family breadth
- trusted retrieval carryover
- stronger long-horizon repo execution
- live non-`vllm` runtime independence

In short:

`the machine has a narrow A4 crossing anchor, but repeated yield and broader
proof are still too weak`

### 3. Proof gaps versus implementation gaps

The current repo state is mixed:

- some gaps are mostly implementation gaps
- some are mostly proof gaps
- some are both

Current classification:

- `retained conversion`: mostly proof gap for broad repeated operation; the
  narrow A4 route has a retained anchor, but fresh integrated campaigns have
  not repeated it reliably
- `trust breadth`: proof gap with some task-supply and supervisor-shaping work
  still needed
- `retrieval carryover`: both implementation and proof gap
- `long-horizon execution quality`: mainly implementation gap
- `decoder independence`: mainly proof gap
- `semantic verifier / stronger memory abstraction / stronger world model /
  deeper search`: substantive implementation gaps

## What To Attack Next

If the goal is the shortest path to a stronger claim, the next work should stay
ordered like this:

1. package the `r6` retained A4 crossing evidence into the canonical proof
   surface instead of regenerating another full `transition_model` candidate
2. run a bounded `r6` replay or evidence-summary validation if a fresh official
   A4 artifact is required
3. get counted breadth across `integration`, `project`, `repository`, and
   `repo_chore` on the credited proof path
4. show retrieval or Tolbert carryover changing later repair or verification
   behavior
5. then push stable persistent queue, resume, and delegated multi-job operation
   for the A4 -> A5 lane

## Minimum Fresh Proof Needed

The smallest integrated rerun that materially upgrades the proof state should
show all of the following in one artifact set:

- `runtime_managed_decisions >= 1`
- `retained_gain_runs >= 1`
- at least one `decision_owner=child_native` with `closeout_mode=natural`
- no `partial_timeout_evidence_only` closeout on the credited proof path
- nonzero counted breadth for `integration`, `project`, `repository`, and `repo_chore`

If the repo also wants the stronger decoder-independence claim, the same run
should additionally show a retained or at least decision-bearing child start on
`provider=hybrid` or another non-`vllm` path.

Source:
[`live_evidence_proof_gap_audit.md`](/data/agentkernel/docs/live_evidence_proof_gap_audit.md:134)

For a narrow A4 confirmation rather than a stronger integrated upgrade, the
smaller proof packet can be limited to the retained `r6` route:
`final_state=retain`, `decision_owner=child_native`,
`decision_conversion_state=runtime_managed`, `closeout_mode=natural`, counted
required-family breadth, and no regressed trace or trajectory tasks.
